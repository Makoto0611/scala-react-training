# レビューファイル作成ルール

## 目的

レビューファイルは「何を直したか」ではなく、**「なぜそうなるのか」を後から読んで理解できる**ことを目的とする。
Step完了後に読み返したとき、コードの意図と修正理由が自分の言葉で説明できる状態を目指す。

---

## 配置ルール

```
scala-react-training/
└── review/
    └── phase1-scala/
        ├── step1-review.md
        ├── step2-review.md
        ├── step3-review.md   ← stepごとに1ファイル
        └── ...
```

- `review/` はプロジェクトルート直下に置く（docs の中に埋めない）
- ファイル名は `stepN-review.md` の形式で統一する

---

## 書き方ルール（必須）

### ① 修正前・修正後を必ずセットで書く

修正後だけを書いても「何がまずかったのか」が伝わらない。
必ず Before / After の両方を書く。

```markdown
### 項目名

#### 修正前

コードと「何が問題か」の説明

#### 修正後

コードと「なぜこの書き方が正しいのか」の説明
```

### ② 「なぜ」を必ず書く

コードの変更だけ書いて終わりにしない。
以下の観点で「なぜ」を説明する：

- なぜ修正前のコードがまずいのか（動くけどまずい理由を含む）
- なぜ修正後の書き方が正しいのか
- Scalaの言語仕様・型システムとどう関係しているのか

### ③ 重要度を明示する

```markdown
**重要度：高** → 必ず直す。設計や型の考え方に関わる
**重要度：中** → できれば直す。将来の問題につながる
**重要度：低** → 理解できればよい。動作には影響しない
```

### ④ 見出し構成を統一する

```markdown
## Part1：〇〇（Stepの課題単位で区切る）

### テーマ名（修正・学習した内容の名前）

#### 修正前のコード

\`\`\`scala
// コード
\`\`\`

なぜこれが問題なのか（箇条書きでも文章でも可）

#### 修正後のコード

\`\`\`scala
// コード
\`\`\`

なぜこの書き方が正しいのか
```

---

## 書かなくてよいもの

- 「できた」「正しい」だけのコメント（改善点がないなら省略してよい）
- 作業ログ（日時・PR番号などは冒頭にまとめる程度でよい）
- 完成度の表（○×の一覧だけでは読み返しの価値がない）

---

## チェックリスト（作成後に確認）

- [ ] 修正前・修正後のコードが両方書かれているか
- [ ] 「なぜ問題なのか」が書かれているか
- [ ] 「なぜ修正後が正しいのか」が書かれているか
- [ ] Scalaの言語仕様・型の話につながる説明があるか
- [ ] 読み返したとき、コードを見ずに説明できそうか
